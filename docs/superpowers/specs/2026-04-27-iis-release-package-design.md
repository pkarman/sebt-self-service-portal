# IIS Release Package — Design

**Date:** 2026-04-27
**Status:** Draft, awaiting user review
**Scope:** DC only (generalize later if CO follows)

## Problem

We're macOS / Linux developers building software that ultimately ships to a state-agency Windows Server / IIS environment. The receiving server admin does not have direct database access — DACPAC application is performed by a separate DBA via ticket. We need our CI to produce a single, self-contained release artifact that contains everything the IIS admin needs (API, web, dacpac, deployment instructions, schema-change summary) without forcing the admin to run pnpm, dotnet, or Node-side build tooling.

The new scripts already added under [scripts/ci/](../../../scripts/ci/) handle the hard parts (dacpac extraction from EF migrations, packaging the Next.js standalone build into a portable, symlink-free zip). What's missing is the orchestration: a CI workflow that runs them in order, produces a deploy-ready bundle, and includes a generated README with version-specific deployment instructions and a dacpac change summary.

## Goals

1. CI produces `sebt-dc-iis-{version}.zip` containing API publish output, web bundle, dacpac, schema delta report, and deployment README.
2. Workflow runs on `workflow_dispatch` (manual rebuilds) and on push of tags matching `release/dc-v*` (versioned releases).
3. README clearly directs the admin to **stop and open a DBA ticket** before applying any database changes; provides a copy-pasteable ticket template.
4. DACPAC change summary (`CHANGELOG-DACPAC.md`) compares against the previous `release/dc-v*` GitHub Release asset; first run emits an "initial release" message instead of failing.
5. Package contents work under the IIS hosting model already chosen: HttpPlatformHandler runs the Next.js `node server.js`; ASP.NET Core Module hosts the .NET API.

## Non-goals

- Refactoring `state-ci.yaml` or consolidating the two parallel CI script directories ([.github/workflows/scripts/](../../../.github/workflows/scripts/) vs [scripts/ci/](../../../scripts/ci/)). Tracked separately.
- Auto-applying the DACPAC. DBA owns that.
- Generalizing for CO. Same shape will work; defer until CO ships.
- Bundling Node.js or the .NET runtime in the zip. Admin installs prerequisites from the standard list (one-time per server).
- Deploy automation. Deliverable is package + README; admin runs the steps.

## Architecture

### New workflow: `.github/workflows/release-iis-dc.yaml`

Triggers:

- `workflow_dispatch` with optional `version` input (defaults to `dev-{short_sha}`)
- `push` of tags matching `release/dc-v*` (version parsed from tag)

High-level steps:

1. Resolve version from tag (`release/dc-v1.2.3` → `1.2.3`) or `workflow_dispatch` input.
2. Check out portal repo and DC state-connector repo (reuse the ref-resolution pattern from [state-ci.yaml:103-124](../../../.github/workflows/state-ci.yaml#L103-L124)).
3. Setup toolchain: .NET 10, Node 24, pnpm 10, `Microsoft.SqlPackage` global tool, `dotnet-ef` global tool, `gh` CLI (preinstalled on `ubuntu-latest`), Docker (preinstalled).
4. Build state connector NuGet package (requires the `build_state_connector_package` function — see "Required script patches" below).
5. Publish API for IIS via new `publish-api.sh`.
6. Build + package frontend via existing [build-frontend.sh](../../../.github/workflows/scripts/build-frontend.sh) and [package-frontend.sh](../../../scripts/ci/package-frontend.sh).
7. Extract DACPAC via existing [extract-dacpac.sh](../../../scripts/ci/extract-dacpac.sh).
8. Generate DACPAC delta report via new `generate-dacpac-report.sh`.
9. Assemble the release bundle via new `bundle-iis-package.sh`.
10. On tag push: create a GitHub Release and attach the bundle and standalone dacpac.
11. On workflow_dispatch: upload bundle as a workflow artifact (90-day retention).

### Package layout

```
sebt-dc-iis-{version}.zip
├── README.md                            # generated per release
├── CHANGELOG-DACPAC.md                  # schema delta vs previous release tag
├── api/
│   ├── *.exe, *.dll                     # dotnet publish output (win-x64, framework-dependent)
│   ├── plugins-dc/                      # DC connector DLLs
│   ├── appsettings.json
│   ├── appsettings.prod.example.json    # only the DC-specific / secret keys the admin must fill in
│   └── web.config                       # ASP.NET Core Module config (auto-generated by publish)
├── web/
│   ├── web.config                       # HttpPlatformHandler config (committed template)
│   ├── src/SEBT.Portal.Web/server.js
│   ├── .next/, public/, node_modules/   # produced by package-frontend.sh
│   └── logs/.gitkeep                    # dir the web.config writes stdout to
└── dacpac/
    ├── sebt-portal-{version}.dacpac
    ├── deploy-report.xml                # raw sqlpackage DeployReport output
    └── deploy-report.html               # rendered for human reading
```

## Components

### 1. `scripts/ci/publish-api.sh` (new)

Single purpose: produce a Windows / IIS-ready publish of the API plus DC connector plugin DLLs and config templates.

- Runs `dotnet publish src/SEBT.Portal.Api/SEBT.Portal.Api.csproj -c Release -r win-x64 --self-contained false -o <out>/api/`.
- Copies DC connector plugin DLLs from the state-connector build output into `<out>/api/plugins-dc/`.
- Writes `<out>/api/appsettings.prod.example.json` containing **only** the keys the admin must supply to run in DC production: secrets and DC-specific overrides (connection string, IdP client secret, any other values that differ from the defaults in `appsettings.json`). Does **not** duplicate defaults from `appsettings.json` — the admin copies this file to `appsettings.Production.json` and fills it in; ASP.NET Core's configuration system will overlay it on top of the base `appsettings.json`.
- Post-processes the auto-generated `<out>/api/web.config` (the one `dotnet publish` emits for the ASP.NET Core Module) to set `stdoutLogEnabled="true"` and `stdoutLogFile=".\logs\stdout"` so the README's reference to `api/logs/stdout_*.log` is real and the admin gets diagnostic output without manual edits. Creates `<out>/api/logs/.gitkeep`.
- Inputs: `--output <dir>` (required), `--state-connector-dir <dir>` (required), `--configuration <Debug|Release>` (default Release).

### 2. `scripts/ci/generate-dacpac-report.sh` (new)

Single purpose: compare the freshly extracted dacpac against the previous release's dacpac and emit a human-readable delta.

- Inputs: `--dacpac <path>` (the new one), `--output-dir <dir>`, `--prev-tag-pattern <glob>` (default `release/dc-v*`).
- Uses `gh release list --limit 50` to find the most recent matching tag, then `gh release download <tag> --pattern '*.dacpac'` to fetch the prior dacpac.
- Runs `sqlpackage /Action:DeployReport /SourceFile:<new>.dacpac /TargetFile:<prev>.dacpac /OutputPath:<out>/deploy-report.xml`.
- Transforms `deploy-report.xml` → `deploy-report.html` using `sqlpackage`'s built-in transform (or an XSLT we ship; XML structure is documented).
- Writes `CHANGELOG-DACPAC.md` summarizing the report:
  - "Compared against `release/dc-v{prev}`."
  - Bullet list grouped by operation (added tables, altered tables, dropped indexes, etc.).
  - Counts at the top.
- **First-run case (no prior release exists):** emits `CHANGELOG-DACPAC.md` containing "Initial release — full schema (no prior baseline). Object counts: N tables, M indexes, K stored procedures." Skips the DeployReport step. Does not fail.

### 3. `scripts/ci/bundle-iis-package.sh` (new)

Single purpose: assemble the directory tree above and zip it.

- Inputs: `--api-dir`, `--web-zip`, `--dacpac`, `--changelog`, `--deploy-report-xml`, `--deploy-report-html`, `--version`, `--output <zip>`.
- Unzips the web bundle into `web/`.
- Drops the committed `web.config` template into `web/web.config` and the `logs/.gitkeep` placeholder.
- Renders `README.md` from a template (see "README template" below) using `envsubst` or `sed` for variable substitution. Variables: `{VERSION}`, `{DACPAC_FILENAME}`, `{DACPAC_SUMMARY}` (first ~5 lines from CHANGELOG-DACPAC.md), `{BUILD_DATE}`, `{GIT_SHA}`.
- Zips the assembled tree to `--output`.

### 4. README template (`scripts/ci/templates/README.iis.md.tmpl`)

Sections:

1. **What's in this package** — annotated file tree.
2. **Prerequisites on the IIS server** — Windows Server with IIS, ASP.NET Core 10 Hosting Bundle, Node.js 24 LTS, HttpPlatformHandler enabled, URL Rewrite module.
3. **Deployment steps** — extract zip; create IIS site for `web/`; create application under it for `api/`; set Application Pool to "No Managed Code" for `web/` and to .NET CLR for `api/`; copy `api/appsettings.prod.example.json` to `api/appsettings.Production.json` and fill in the DC-specific / secret values (connection string from DBA, IdP secrets, etc.); set ACLs on `web/logs/` and `api/logs/` for the app pool identity.
4. **Database update (DBA ticket)** — bold callout that the IIS admin **does not run the dacpac**. Includes a copy-pasteable ticket template:
   - Subject: `SEBT Portal DC v{VERSION} — DACPAC application`
   - Body: package version, dacpac filename, schema-change summary, link to deploy-report.html, exact `sqlpackage /Action:Publish` command for the DBA.
5. **Verification** — health-check URLs, log file locations (`web/logs/node-stdout*`, `api/logs/stdout_*.log`), expected success indicators.
6. **Rollback** — preserve previous package dir; site swap procedure; DBA-side rollback is a separate ticket.

### 5. `web/web.config` (committed template)

The HttpPlatformHandler config provided by the user, committed to the repo at e.g. `scripts/ci/templates/web.config` and copied into the package by `bundle-iis-package.sh`. No per-release templating needed; environment-specific values (BACKEND_URL, NEXT_PUBLIC_STATE) stay where they are because the package is per-state and the API path is a stable convention.

### 6. Required script patches

`scripts/ci/build-backend.sh` is currently missing the `build_state_connector_package` function present in `.github/workflows/scripts/build-backend.sh:125-153`. Port that function back in so `scripts/ci/build-backend.sh` is self-sufficient when invoked by the new workflow. Keep `.github/workflows/scripts/build-backend.sh` untouched (it's still wired into `state-ci.yaml`).

## Error handling

- Each `scripts/ci/*.sh` exits non-zero on failure with a clear log_error message and an actionable hint (already the convention in the existing scripts).
- `generate-dacpac-report.sh` distinguishes "no prior release found" (write initial-release CHANGELOG, exit 0) from "gh CLI failed" (exit non-zero).
- `extract-dacpac.sh` already handles the temp-DB cleanup via `trap` and is reused as-is.
- The workflow uses `if: failure()` to upload partial artifacts (the dacpac and partial publish output) for diagnosis when the bundle step fails.

## Testing

- Add a smoke test invocation in CI on PRs that touch any of `scripts/ci/{publish-api,generate-dacpac-report,bundle-iis-package}.sh` or `release-iis-dc.yaml`: run the workflow's build-bundle steps end-to-end on `ubuntu-latest`, but skip the GitHub Release publish step. Assert the zip exists, contains all expected paths, and `unzip -l` shows non-zero sizes for each major component.
- For `generate-dacpac-report.sh`, unit-test the first-run branch by running it with `--prev-tag-pattern release/never-matches-*` and asserting the initial-release CHANGELOG is emitted.
- Manually verify on first real run that the produced zip extracts cleanly on a Windows test machine and the IIS deployment instructions in the README work end-to-end.
